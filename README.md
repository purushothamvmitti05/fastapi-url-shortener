# FastAPI URL Shortener with Analytics

A backend API that shortens URLs, redirects visitors to the original link, and tracks click analytics (count, timestamp, referrer) — built to practice production backend and DevOps fundamentals end to end.

**Live demo:** https://fastapi-url-shortener-93bz.onrender.com (UI) · [/docs](https://fastapi-url-shortener-93bz.onrender.com/docs) for the API reference
*(Free-tier hosting — the first request after inactivity may take ~50 seconds to wake up.)*

## What it does

- Shorten any URL into a random 6-character short code
- Simple web UI to paste a link, get a short link, copy it, and check live click counts
- Visiting a short link redirects to the original URL
- Every visit is logged with a timestamp and referrer
- View click analytics for any short code (via UI refresh button or the API directly)
- Rate limiting (5 requests/minute) on URL creation to prevent abuse

## Tech Stack

**Backend:** Python, FastAPI, Pydantic, SQLAlchemy
**Frontend:** HTML, CSS, JavaScript (server-rendered via Jinja2, vanilla JS for API calls)
**Database:** PostgreSQL (SQLite for local fallback)
**DevOps:** Docker, Docker Compose, GitHub Actions (CI), Render (deployment), Linux
**Other:** slowapi (rate limiting)

## Architecture

```
app/
├── main.py          # API routes + homepage route
├── database.py      # DB connection/session setup
├── models.py        # SQLAlchemy table definitions (URL, Click)
├── schemas.py       # Pydantic request/response models
├── crud.py          # Database read/write logic
├── templates/
│   └── index.html   # Frontend homepage (Jinja2 template)
└── static/
    ├── style.css     # Frontend styling
    └── script.js     # Frontend logic (calls the API via fetch)
```

The project follows a layered structure separating routing, data validation, and database logic — making it easier to test, maintain, and extend.

## API Endpoints

| Method | Endpoint              | Description                          |
|--------|------------------------|---------------------------------------|
| GET    | `/health`              | Health check                         |
| POST   | `/shorten`             | Create a short URL (rate-limited)    |
| GET    | `/{short_code}`        | Redirect to the original URL         |
| GET    | `/{short_code}/stats`  | View click analytics for a short URL |

Full interactive API docs available at `/docs` (Swagger UI), auto-generated by FastAPI.

## Running Locally

### With Docker Compose (recommended — includes PostgreSQL)

```bash
git clone https://github.com/purushothamvmitti05/fastapi-url-shortener.git
cd fastapi-url-shortener
docker-compose up --build
```

App will be available at `http://localhost:8000/docs`

### Without Docker (SQLite fallback)

```bash
python -m venv venv
venv\Scripts\Activate.ps1   # Windows
pip install -r requirements.txt
uvicorn app.main:app --reload
```

## CI/CD

Every push to `main` triggers a GitHub Actions workflow that:
1. Installs dependencies in a clean environment
2. Verifies the app imports without errors
3. Builds the Docker image

This catches broken dependencies or import errors before deployment — same pipeline structure used in production engineering workflows.

## What I learned building this

- Designing relational schemas with SQLAlchemy (one-to-many relationships via foreign keys)
- Dependency injection patterns in FastAPI (`Depends`, session lifecycle management)
- Separating API contracts (Pydantic schemas) from database models — a clean architecture pattern
- Containerizing a multi-service app (app + database) with Docker Compose, including persistent volumes
- Debugging real environment issues: a missing dependency that only worked locally by accident, a file-encoding bug (UTF-16 vs UTF-8) that broke `requirements.txt`, and a breaking API change in Starlette's `TemplateResponse` signature that required reading release notes to diagnose
- Building a server-rendered frontend with Jinja2 templates and vanilla JavaScript, talking to the existing API via `fetch` — no frontend framework needed
- Setting up a CI pipeline with GitHub Actions
- Working with Linux — the app runs inside a `python:3.11-slim` Linux container, and debugging (via `docker logs`, container shells) happens on Linux, not Windows
- Deploying a containerized app with a managed PostgreSQL database to a live cloud environment
